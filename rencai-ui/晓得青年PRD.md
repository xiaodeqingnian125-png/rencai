# 晓得青年 · 郑州人才公寓小程序 PRD

> **版本：** V2.4
> **日期：** 2026-06-28
> **作者：** 晓得（产品经理）
> **状态：** 待评审
> **说明：** 文件名"V1.0"指MVP大版本号，文档内"V2.4"指本PRD的第2.4次修订。

## 1. 项目背景

### 产品定位
晓得青年是郑州人才公寓垂直信息服务平台微信小程序，解决官方平台信息分散、更新滞后、体验差的问题，为来郑年轻人提供一站式「找公寓→看活动→找室友→享社区」服务。

### 业务目标
- **3个月内**：小程序日活100+，私域社群300+人
 - **3个月内**：小程序日活300+，私域社群300+人
- **6个月内**：小程序日活500+，月变现覆盖运营成本
- **核心指标**：用户从浏览到加群转化率 > 15%，活动报名转化率 > 8%

### 目标用户
| 用户画像 | 典型特征 | 核心诉求 |
|---------|---------|---------|
| 应届毕业生（22-25岁） | 刚到郑州找工作，对城市陌生 | 快速找到性价比高的住处，认识同频室友 |
| 来郑青年人才（25-30岁） | 已工作1-3年，追求生活品质 | 了解各项目真实评价，参与社区活动 |
| 准备申请人才公寓者 | 在纠结选哪个项目 | 全面对比各项目租金/位置/配套，避免踩坑 |

### 价值主张
- **对用户**：一个入口搞定所有人才公寓信息，不用在5个公众号+3个群里来回翻
- **对晓得青年**：用信息差+自动化工具做本地流量入口，小红书获客→微信私域沉淀→小程序服务变现

### 为什么现在做
1. 郑州人才公寓政策持续推进，2026年新项目持续交付，信息整合需求刚性存在
2. 官方平台体验差是长期痛点，竞品几乎为零，窗口期明确
2. 官方平台体验差是长期痛点，窗口期明确
3. 小红书+影刀自动化内容生产能力已跑通，获客管道就位，缺承接载体

### 竞品分析
| 竞品/替代方案 | 类型 | 劣势 |
|-------------|------|------|
| 郑州市住建局官网 | 官方平台 | 信息更新慢，无移动端适配，无法结构化筛选对比 |
| 各公寓项目公众号 | 分散渠道 | 每个项目独立运营，用户需关注5+个号来回切换 |
| 小红书/微信群 | UGC社区 | 信息碎片化、真伪难辨、无结构化数据 |
| 58同城/贝壳 | 租房平台 | 聚焦普通租房，无人才公寓分类，无社区/找室友属性 |
| 郑好办App | 政务平台 | 侧重申请流程，无找室友/活动/社区功能 |

## 2. 需求概述

### 用户痛点
1. **信息碎片化**：人才公寓信息散落在住建局官网、各项目公众号、物业群、小红书帖子，想对比要打开10个页面
2. **信息滞后**：官方平台房源状态更新慢，空房信息常过期，白跑一趟
3. **缺乏真实评价**：没有统一渠道看到已入住用户的真实体验和避坑指南
4. **社交孤立**：住进人才公寓后，不认识邻居，缺乏社区归属感
5. **找室友难**：有房想找人合租，或者想找人一起申请，缺少靠谱的匹配渠道

### 业务价值
| 价值维度 | 说明 |
|---------|------|
| 用户价值 | 信息一站式获取，降低决策成本；社区活动增加归属感；找室友降低居住成本 |
| 平台价值 | 沉淀本地精准流量，构建「内容→社群→服务」闭环；数据资产可延伸至票务、代运营等变现 |
| 生态价值 | 连接公寓项目方、周边商家、活动主办方，做本地生活服务入口 |

### 预期收益（V1.0 MVP阶段）
| 指标 | 第1个月 | 第3个月 | 第6个月 |
|------|---------|---------|---------|
| 累计注册用户 | 500-800 | 3,000-5,000 | 10,000+ |
| 日活用户 | 50-80 | 300-500 | 1,000+ |
| 社群数量 | 5-8个 | 15-20个 | 30+个 |
| 月活动场次 | 3-5场 | 8-12场 | 15-20场 |

### 版本范围
**V1.0 MVP**：聚焦「房源信息展示（含户型）+ 活动发布报名 + 找室友 + 借个锤子（物品共享）+ 评论 + 消息通知」六大核心功能，验证需求闭环。会员积分体系、数据看板等放V2.0。

## 4. 业务流程

### 4.1 核心用户路径

```
首次进入小程序
  → 首页（公寓卡片列表，游客可直接浏览）
  → 点击公寓卡片 → 触发登录授权 → 公寓详情（价格/位置/配套/周边/户型卡片+评论）
    → 点击户型卡片 → 户型详情（图片/户型介绍/朝向/价格/配套/评论）
  → TabBar切换：服务 / 借个锤子 / 消息 / 我的（均需登录）
  → 完成目标动作（浏览/评论/报名/发帖/借物品/加群）
```

### 4.2 公寓浏览流程

#### 正常流程
1. 用户进入首页，看到公寓推荐卡片列表（默认按热度排序）
2. 用户可通过顶部筛选（区域/价格）缩小范围
3. 点击公寓卡片进入公寓详情页
4. 查看价格区间、位置、配套设施、周边服务
5. 浏览该公寓下属的户型卡片列表
6. 点击户型卡片进入户型详情页
7. 查看图片、户型介绍、朝向、价格区间、配套设施
8. 在公寓详情页或户型详情页发表/查看评论
9. 点击「收藏」或「分享」

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 筛选无结果 | 提示「暂无符合条件的公寓，试试放宽筛选条件？」+ 推荐相邻区域 |
| 公寓详情加载失败 | 显示骨架屏+重试按钮，3秒内未加载则提示「加载失败，点击重试」 |
| 图片加载慢 | 先显示低清占位图，高清图懒加载 |
| 该公寓暂无户型数据 | 显示「户型信息整理中，敬请期待」 |

### 4.3 活动报名流程

#### 正常流程
1. 用户进入活动列表页，浏览近期活动
2. 点击感兴趣的活动进入详情页
3. 查看时间、地点、费用、人数限制、活动标签（官方/用户）
4. 点击「立即报名」
5. 确认报名信息（姓名、手机号默认读取微信授权信息；若用户未授权手机号则降级为手动输入）
6. 报名成功，显示成功页
7. 报名成功后引导用户授权订阅消息，活动前2小时收到订阅消息提醒

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 活动名额已满 | 按钮变灰「已满员」，展示候补提醒文案，引导用户关注后续名额或其他活动；V1.0 不生成候补报名记录 |
| 重复报名 | 提示「您已报名该活动」，显示活动详情 |
| 报名截止 | 按钮变灰「报名已截止」，推荐其他可报名活动 |
| 活动取消 | 已报名用户收到取消通知，活动状态标记「已取消」 |

### 4.3.1 用户发起活动流程

#### 正常流程
1. 用户点击活动页「发起活动」按钮
2. 填写活动信息：标题、封面图、类型（线上/线下）、时间、地点、费用、人数限制、活动介绍
3. 提交活动，进入管理员审核队列
4. 管理员后台审核通过 → 活动发布上线，展示「用户活动」标签
5. 管理员审核不通过 → 通知用户修改原因
6. 活动上线后，发帖人即为「活动主办」，可在个人中心查看
7. 活动若设置费用，报名成功页展示费用金额+主办方联系方式，用户线下转账；报名确认不以付款为条件

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 活动信息不完整 | 必填项校验不通过，高亮缺失字段 |
| 封面图违规 | 管理员审核拒绝，通知用户更换 |
| 活动内容不合规 | 管理员审核拒绝，说明原因 |
| 发起人频繁提交 | 单用户每日最多发起2个活动 |

### 4.4 找室友（住搭子）流程

#### 正常流程
1. 用户进入「住搭子」板块
2. 选择「有房找室友」或「无房找合租」
3. **如果选择「有房找室友」**：
   - 必须确认已抢到房源（勾选「我已获得该公寓房源」）
   - 绑定公寓项目（下拉选择）
   - 填写房源详情：户型、租金、楼层、朝向、面积等
   - 帖子自动展示「已确认房源」标签，增强可信度
4. **如果选择「无房找合租」**：
   - 填写期望公寓/预算范围/性别偏好
5. 补充个人简介/入住时间/联系方式
6. 发布成功，帖子进入审核队列 → 审核通过后出现在列表
7. 其他用户浏览帖子，点击「联系TA」
8. 用户点击「联系TA」→ 弹出客服二维码（与加入社群共用同一助理微信），长按识别添加，备注帖子ID，由客服协助对接

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 发布内容违规 | 管理员后台审核不通过，通知用户修改 |
| 「有房找室友」未确认房源 | 强制要求勾选确认+填写房源信息，否则无法发布 |
| 帖子过期（30天未更新） | 帖子到期自动下架，系统发消息提醒发帖人「您的找室友帖子已到期下架，如需继续请重新发布」 |
| 无人响应 | 系统推送「相似需求推荐」，推荐匹配度高的帖子 |

### 4.5 借个锤子（物品共享）流程

#### 正常流程
1. 用户进入「借个锤子」Tab页，浏览可借用物品列表
2. 可按品类筛选（工具/户外/小家电等）
3. 点击物品卡片查看详情（物品描述、图片、物主信息、借用规则）
4. 点击「申请借用」，选择借用时间+留言
5. 借用申请发送至物主，物主在消息页收到通知
6. 物主确认借用 → 可选填写具体房间号/交接位置，随确认消息发送给借用人
7. 借用人收到确认；若物主填写了房间号/交接位置则一并展示，双方约定交接方式；若未填写，引导借用人在消息页主动联系物主协商
8. 借用完成，物主确认归还

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 物品已被借出 | 显示「借出中，预计X号归还」，提供「预约借用」选项 |
| 物主拒绝借用 | 通知申请人「物主暂时无法出借」 |
| 借用超时未还 | 系统提醒借用人+通知物主，超7天标记「逾期」，消息页推送逾期提醒给双方 |
| 物品损坏纠纷 | 引导双方协商，管理员介入调解 |

### 4.6 评论流程

#### 正常流程
1. 用户在公寓详情页或户型详情页底部查看评论列表
2. 点击「写评论」按钮
3. 输入评论内容（文字，限制200字，无评分）
4. 提交评论，直接展示（无需审核）
5. 其他用户可查看评论，可对评论点赞
6. 管理员可在「我的」→「评论管理」中对违规评论进行隐藏/删除

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 评论内容违规 | 管理员在评论管理中隐藏/删除，无需前端审核 |
| 重复提交 | 前端防抖，同一用户对同一对象5分钟内只能评论1次 |
| 评论对象已下架 | 评论仍保留但不公开展示 |

### 4.7 代办服务流程

#### 正常流程
1. 用户进入服务页，点击「代办服务」
2. 浏览服务列表（代办手续/代排号/代取件等）
3. 点击服务卡片进入详情页，查看服务说明+价格
4. 点击「下单」，填写订单信息（联系人/电话默认读取用户授权手机号，若未授权则降级为手动输入；支持修改/地址/备注）
5. 确认订单信息+金额，点击「支付」
6. 完成微信支付，订单状态变更为「已支付」
7. 管理员/服务人员接单处理，状态变更为「处理中」
8. 服务完成，状态变更为「已完成」

#### 异常流程
| 异常场景 | 处理方式 |
|---------|---------|
| 支付失败 | 提示「支付失败，请重试」，订单保留在「待支付」状态 |
| 用户取消订单 | 「待支付」状态可直接取消；「已支付」需联系客服退款 |
| 服务超时未完成 | 系统提醒管理员，超24小时标记「超时」 |
| 退款申请 | 引导联系客服处理，1-3个工作日退款 |

### 4.8 状态流转

### 4.7.1 代办服务降级方案

> **说明：** 若微信支付商户号在 Phase 2 启动前仍未落地，代办服务下单按钮改为「提交需求」，订单仅记录不发起支付，由客服后续联系确认并线下完成交易。待商户号就绪后再上线在线支付。

```
用户状态：游客 → 注册用户 → 活跃用户 → 核心用户
帖子状态：待审核 → 已发布（含「已确认房源」标签或「寻找合租」标签）→ 已过期 / 已下架（到期自动下架+消息提醒）/ 已拒绝
活动状态：待审核 → 报名中 → 已满员 → 进行中 → 已结束 / 已取消
报名状态：已报名 → 已取消
物品状态：可借用 → 借出中 → 已归还 / 已下架
借用状态：申请中 → 已确认 → 借用中 → 已归还 / 已逾期 / 已拒绝
评论状态：已展示 / 已隐藏（管理员可隐藏违规评论）
消息状态：未读 → 已读
服务订单状态：待支付 → 已支付 → 处理中 → 已完成 / 已取消 / 超时
```

### 6.3 技术方案

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端框架 | Taro + React + TypeScript | 跨端适配微信小程序，全栈TS统一语言 |

| 后台管理（Web端） | React + TypeScript + Tailwind CSS | 与小程序共用Supabase项目，数据打通；处理公寓/户型/服务编辑、订单管理等数据录入类操作 |
| 部署 | 微信小程序审核上线 + Supabase免费额度 + 微信支付商户号 | 最小启动成本约350元（不含支付押金） |
